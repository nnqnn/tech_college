# Отчет по реализации третьего этапа

## Работоспособность проекта

Проект можно запустить и проверить на практике. Связка реализованных сервисов такая:

1. Telegram-бот принимает команды и сообщения пользователя.
2. Бот по HTTP обращается в Backend API.
3. Backend API сохраняет пользователей, анкеты, взаимодействия и рейтинги в PostgreSQL.
4. Backend API использует Redis для кэширования пачки следующих анкет.
5. Backend API публикует события `FeedRequested` и `InteractionCreated` в RabbitMQ.
6. Worker читает события из RabbitMQ и пишет их в лог.
7. При лайке или пропуске Backend обновляет взаимодействия, рейтинг и очищает кэш пользователя.

Для запуска нужны:

- PostgreSQL
- Redis
- RabbitMQ
- Backend API на FastAPI
- Worker очереди
- Telegram-бот с реальным `TELEGRAM_BOT_TOKEN`

Команды запуска:

```bash
source .venv/bin/activate
pip install -r requirements.txt
docker compose up -d
uvicorn backend.main:app --reload
```

В отдельном терминале:

```bash
source .venv/bin/activate
python -m worker.main
```

Еще в одном терминале:

```bash
source .venv/bin/activate
python -m bot.main
```

Если на компьютере установлен старый Docker Compose:

```bash
docker-compose up -d
```

Если Redis уже занят на порту `6379`, можно поднять только Postgres:

```bash
docker-compose up -d postgres
```

Если backend падает с ошибкой подключения к `127.0.0.1:5432`, значит PostgreSQL не запущен. Нужно поднять Postgres через Docker Compose и снова запустить backend.

## Что выполнено по ТЗ

### CRUD для анкет

Реализованы backend-эндпоинты:

- `PUT /api/v1/profiles/{telegram_id}` - создать или обновить анкету.
- `GET /api/v1/profiles/{telegram_id}` - получить анкету.
- `GET /api/v1/profiles` - получить список анкет.
- `DELETE /api/v1/profiles/{telegram_id}` - очистить анкету пользователя.

Поля анкеты:

- возраст;
- пол;
- интересы;
- город;
- возрастные предпочтения;
- предпочтительный пол;
- предпочтительный город;
- предпочтения по интересам;
- количество фото;
- реферальный Telegram ID.

Основная реализация находится в:

- `backend/main.py`
- `backend/schemas.py`
- `backend/storage.py`

В Telegram-боте CRUD теперь доведен до пользовательского интерфейса:

- при нажатии `📝 Моя анкета` пользователь сначала видит свои текущие данные;
- если анкеты нет, бот сообщает, что она не заполнена;
- бот предлагает `✏️ Заполнить заново`;
- если анкета есть, бот дополнительно предлагает `🗑 Удалить анкету`;
- при заполнении анкеты есть клавиатурные подсказки для пола и предпочтительного пола.

Основная реализация находится в:

- `bot/api_client.py`
- `bot/handlers.py`
- `bot/keyboards.py`

### Система ранжирования

Рейтинг хранится отдельно в таблице `user_ratings`.

Эндпоинт получения рейтинга:

- `GET /api/v1/users/{telegram_id}/rating`

Реализованы три уровня рейтинга.

Уровень 1, первичный рейтинг:

- учитывает заполненность анкеты;
- учитывает возраст, пол, интересы и город;
- учитывает предпочтения пользователя;
- учитывает количество фото.

Уровень 2, поведенческий рейтинг:

- учитывает количество полученных лайков;
- учитывает соотношение лайков и пропусков;
- учитывает взаимные лайки;
- учитывает свежесть активности пользователя.

Уровень 3, комбинированный рейтинг:

- объединяет primary, behavioral и referral scores;
- использует формулу:

```text
total_score = 0.55 * primary_score + 0.35 * behavioral_score + 0.10 * referral_score
```

- учитывает реферальный фактор через `referral_telegram_id`.

Основная реализация находится в:

- `backend/ranking.py`
- `backend/storage.py`

### Лайки, пропуски и мэтчи

Реализован эндпоинт:

- `POST /api/v1/interactions`

Он принимает:

- `requester_telegram_id`;
- `responder_telegram_id`;
- `is_like`.

Поведение:

- лайк или пропуск сохраняется в `user_interactions`;
- повторная оценка той же анкеты возвращает `409 Conflict`;
- если оба пользователя поставили друг другу лайк, возвращается `match: true`;
- после взаимодействия пересчитывается рейтинг;
- после взаимодействия очищается Redis-кэш пользователя.
- после взаимодействия публикуется MQ-событие `InteractionCreated`.

Основная реализация находится в:

- `backend/main.py`
- `backend/storage.py`

### Выдача анкет и Redis-кэш

Реализован эндпоинт:

- `GET /api/v1/feed/{telegram_id}/next`

Логика:

1. Backend сначала пытается взять следующую анкету из Redis.
2. Если кэш пуст, Backend собирает пачку кандидатов из PostgreSQL.
3. Кандидаты фильтруются по предпочтениям пользователя.
4. Уже оцененные анкеты не показываются повторно.
5. Сам пользователь не попадает в свою выдачу.
6. Кандидаты сортируются по `total_score`.
7. Первая анкета возвращается пользователю, остальные кладутся в Redis.
8. При выдаче анкеты публикуется MQ-событие `FeedRequested`.

Формат ключа Redis:

```text
candidates:{telegram_id}
```

Основная реализация находится в:

- `backend/cache.py`
- `backend/main.py`
- `backend/storage.py`

### PostgreSQL

Backend при старте автоматически создает таблицы:

- `users`;
- `user_photos`;
- `user_interactions`;
- `user_ratings`.

Также создаются индексы для быстрых выборок по пользователям, взаимодействиям и рейтингу.

Основная реализация находится в:

- `backend/storage.py`

Инфраструктура для локального запуска:

- `docker-compose.yml`
- `.env.example`

### RabbitMQ

Добавлен MQ-сервис RabbitMQ.

Backend публикует события:

- `InteractionCreated`;
- `FeedRequested`.

События отправляются в очередь:

```text
dating.events
```

Worker запускается командой:

```bash
python -m worker.main
```

Он читает очередь и выводит полученные события в лог. Это закрывает потоковую обработку событий и связь Backend с MQ.

Основная реализация находится в:

- `backend/events.py`
- `worker/main.py`
- `docker-compose.yml`

## Что не добавлялось в этом этапе

В документации описаны дополнительные сервисы и направления развития:

- Celery worker;
- S3/MinIO;
- Prometheus/Grafana.

Celery, S3/MinIO и Prometheus/Grafana пока не добавлялись. MQ добавлен через RabbitMQ: Backend публикует события, worker их читает и логирует. Пересчет рейтинга пока остается синхронным в Backend, чтобы поведение бота сразу было понятно при показе.
