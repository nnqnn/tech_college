# Описание сервисов

Ниже описаны компоненты системы и их ответственность. Telegram-бот обязателен как внешний интерфейс, но его реализация не входит в текущий объём.

## Telegram Bot (внешний интерфейс)
- Принимает команды пользователя (анкета, просмотр, лайк/пропуск).
- Отправляет запросы в Backend по HTTP (webhook/long polling — не принципиально для дизайна).

## Backend API (FastAPI)
- Единая точка входа для Telegram-бота.
- Управляет анкетой: создание/обновление профиля, предпочтения, активность.
- Отдаёт следующую анкету для просмотра (с учётом фильтров и ранжирования).
- Принимает события взаимодействий (лайк/пропуск/взаимный лайк) и публикует их в очередь.
- Работает с хранилищем фото (S3/MinIO): выдаёт ссылки на загрузку/получение.
- Экспортирует метрики (Prometheus) и пишет структурированные логи.

## PostgreSQL (основное хранилище)
- Анкеты пользователей, предпочтения.
- История взаимодействий.
- Текущие значения рейтингов (в отдельной таблице).
- Минимально необходимые индексы для быстрых выборок.

## Очередь сообщений (RabbitMQ/Kafka)
- Поток событий взаимодействий с анкетами.
- Развязка Backend и асинхронных обработчиков (пересчёт поведенческого рейтинга, статистика).

## Celery Worker (пересчёты/фоновая обработка)
- Регулярно пересчитывает рейтинги пользователей и записывает результат в таблицу рейтингов.
- Может поддерживать подготовку «кандидатных списков» для кэша (опционально).

## Redis (кэш кандидатов)
- Кэширует предварительно отранжированные списки анкет на пользователя (например, по 10 штук).
- Обновляется «пачками»: первая анкета считается «полным путём», параллельно догружаются следующие.

## S3/MinIO (фото)
- Хранит изображения анкет.
- Backend хранит только ключи/метаданные; сами файлы — в объектном хранилище.

## Метрики и наблюдаемость (Prometheus + Grafana)
- Prometheus собирает метрики backend/worker/очереди.
- Grafana — дашборды для администратора (ошибки, задержки, скорость событий, пересчёты рейтинга).
